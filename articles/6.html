Introduction

Cloudflare has become a cornerstone for many companies looking to improve their website performance and security. As described in Cloudflare's blog post, when a user signs up for Cloudflare, they attached with two unique name servers, each named after a boy and a girl. These name servers, such as bob.ns.cloudflare.com and lola.ns.cloudflare.com, play a crucial role in the seamless integration of Cloudflare‚Äôs services with the user‚Äôs domain.

Cloudflare‚Äôs design of name servers aims to simplify the sign-up process and ensure smooth DNS resolution. However, this naming convention can also be leveraged for security research and asset discovery.

Discovering Cloudflare-Protected Assets

The unique pairing of Cloudflare name servers for each account offers an interesting opportunity to identify related domains under the same Cloudflare account. Here‚Äôs a step-by-step guide on how to uncover these assets:

  1. Identify the Cloudflare Name Servers:
    Start by examining the DNS records of a target domain to identify its Cloudflare name servers. These will typically follow the format
    [boy].ns.cloudflare.com and [girl].ns.cloudflare.com.
  1. Reverse Lookup for Domain Associations:
    Perform a reverse NS lookup to identify other domains using the same name servers. The chances of multiple accounts having the same pair of name servers are very slim, making this a reliable method to uncover related domains.
  1. Cross-Reference Name Servers:
    By cross-referencing the results of the reverse lookups for both name servers, you can pinpoint domains that are highly likely to be associated with the same Cloudflare account.

Practical Implementation

While in theory it sounds simple, reverse NS lookups are not widely available through free or standard services. However, platforms like SecurityTrails offer the necessary tools. SecurityTrails provides an API that allows for reverse DNS lookups, albeit with a limit of 50 API requests per month (as of the time of writing this article).

SecurityTrails Specifics:

  • SecurityTrails allows us to identify how many domains have NS records pointing to a specific Cloudflare domain.
  • SecurityTrails gives you a maximum of 10K results per search.
  • These 10K results are the top-ranked sites associated with the name server. It's important to note that usually only a few hundred domains have a rank, while the rest are randomized from the pool of unranked domains.
  • As a result, the cross-referencing between two name servers will never be a complete list, as only the top 10K sites from each name server are considered.

To address this limitation, I added an additional feature from SecurityTrails to my script. This feature allows for keyword searches within the full results, enabling queries for domains that might not be in the top 10K but are still relevant to the search criteria.

Manual Flow of the Technique:

  • First, let's choose a target for our reconnaissance efforts. We will take onezerobank.com, a bank based in Israel that protects its domains with Cloudflare.
  • Then, we can follow the two lists for each NS and filter by keywords. We can use keywords associated with the target, such as bank, onezero, and .co.il (Israel extension).
  • Now we just need to cross-reference the two lists for every keyword. Here is the output:
    • digibank.co.il
    • onezerobank.co.il
    • onezerobank.com
    • tfd-bank.com
